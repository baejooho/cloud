## 📌 PART 1. Git 초급

### 1-1. Git 3단계 영역

```
Working Directory  →(git add)→  Staging Area  →(git commit)→  Repository
   내가 작업하는 곳              커밋 대기 공간              영구 저장소
```

### 1-2. 파일 생명주기 4가지 ★★★

| 상태 | 설명 |
|------|------|
| **Untracked** | git add를 한 번도 안 한 상태. Git이 추적 안 함 |
| **Unmodified** | add 후 commit 완료, 수정 없음 (= new file) |
| **Modified** | commit 이후 파일을 수정한 상태 |
| **Staged** | git add 후 Staging Area에 올라간 상태 |

### 1-3. 기본 명령어

```bash
git init                    # 저장소 초기화
git status                  # 현재 상태 확인
git add <파일>              # Staging Area로 이동
git add .                   # 전체 파일 add
git commit -m "메시지"      # 커밋
git log                     # 커밋 내역 확인
git log --all --oneline --graph  # 그래프 형태로 확인
```

### 1-4. reset 3가지 옵션 ★★★ (자주 출제!)

| 옵션 | 이동 위치 | 파일 상태 |
|------|-----------|-----------|
| `--soft` | Staging Area | commit 직전 상태 |
| `--mixed` | Working Directory | add 직전 상태 **(기본값)** |
| `--hard` | 완전 삭제 | 흔적도 없이 제거 ⚠️ |

```bash
git reset --soft [커밋ID]
git reset --mixed [커밋ID]
git reset --hard [커밋ID]
```

### 1-5. reset vs revert ★★★

| | reset | revert |
|--|-------|--------|
| 방식 | 원하는 시점으로 돌아가고 이후 커밋 **삭제** | 되돌리는 **새 커밋 추가** |
| 기록 | 삭제됨 | **보존됨** |
| 협업 | 위험 | **안전** |

> **💡 핵심:** 협업에서는 revert 사용! reset은 개인 작업에서만!

## 📌 PART 2. Git 브랜치 (중급)

### 2-1. 브랜치 명령어 6개 ★★★

```bash
git branch                      # 브랜치 목록 확인
git branch <이름>               # 브랜치 생성
git switch <이름>               # 브랜치 이동
git switch -c <이름>            # 생성 + 이동 동시에 ★
git branch -m <기존> <새이름>   # 브랜치 이름 수정
git branch -d <이름>            # 브랜치 삭제
```

### 2-2. merge vs rebase ★★★

| | merge | rebase |
|--|-------|--------|
| 방식 | 두 브랜치 합쳐서 **새 커밋 생성** | 커밋을 다른 브랜치 위에 **재배치** |
| 그래프 | 분기 형태 (갈라짐) | 히스토리 **일직선** |
| 명령어 | `git merge <브랜치명>` | `git rebase <브랜치명>` |

### 2-3. 충돌(Conflict) 해결 방법 ★★

**merge 충돌:**
```bash
# 취소
git merge --abort
# 해결 순서
파일 수정 → git add → git commit
```

**rebase 충돌:**
```bash
# 취소
git rebase --abort
# 해결 순서
파일 수정 → git add → git rebase --continue
```

> **충돌 표시 읽는 법:**
> ```
> <<<<<<< HEAD          ← 현재 브랜치 내용
> 내 코드
> =======
> 상대방 코드
> >>>>>>> 브랜치명      ← 합치려는 브랜치
> ```

## 📌 PART 3. Git 고급 (GitHub 연동)

### 3-1. push / pull 개념

```
로컬 저장소  →(git push)→  GitHub(원격)
로컬 저장소  ←(git pull)←  GitHub(원격)
```

### 3-2. 최초 push 3단계 순서 ★★★

```bash
# 1단계: 원격 저장소 연결
git remote add origin https://github.com/계정/레포.git

# 2단계: 브랜치 이름 main으로 설정
git branch -M main

# 3단계: 최초 push (-u로 upstream 설정)
git push -u origin main
```

> 이후 push는 그냥 `git push` 만 해도 됨!

### 3-3. push 전 pull 필수! ★★★

```bash
git pull origin main    # ← 반드시 먼저!
git add .
git commit -m "메시지"
git push origin main    # ← 나중에!
```

> **이유:** 다른 사람의 변경사항을 먼저 받아와야 충돌 예방 가능

### 3-4. clone vs pull

| | git clone | git pull |
|--|-----------|----------|
| 사용 시점 | **최초 1회**, 전체 복사 | 이미 있는 로컬에 **변경사항만** |
| 명령어 | `git clone <URL>` | `git pull origin main` |

### 3-5. Access Token ★

- GitHub는 **2021년 8월부터** 비밀번호 인증 제거
- push 시 password 자리에 **Access Token** 입력 필요
- 오류: `Authentication failed` → Access Token 미설정이 원인

### 3-6. 다른 브랜치에 push

```bash
git switch leader
git push -u origin leader
```

## 📌 PART 4. AWS 핵심 구조

### 4-1. AWS 계층 구조 (크기 순서) ★★★

```
리전(Region)
  └── 가용영역(AZ, Availability Zone)
        └── VPC (Virtual Private Cloud)
              └── 서브넷(Subnet)
                    └── EC2 인스턴스
```

### 4-2. 리전(Region)

- AWS 서비스가 제공되는 **물리적 위치**
- 각 리전에는 고유 코드 부여
  - 서울: `ap-northeast-2`
  - **버지니아 북부: `us-east-1` ← 모든 AWS 서비스 최초 출시 리전 ★**

**리전 선택 기준:**
1. **지연속도** - 사용자와 가까운 리전 선택
2. **법률** - 개인정보는 해당 국가 서버에 저장 의무
3. **서비스 가용성** - 리전마다 제공 서비스가 다름

### 4-3. 가용영역(AZ, Availability Zone)

- 리전의 하부 단위
- 하나의 리전 = **2개 이상**의 가용영역으로 구성
- 1개 이상의 물리 데이터센터를 묶은 논리적 데이터센터
- AZ 간 거리: **반드시 몇 km 이상** 떨어져 있음 (재해 대비)
- 단, 모든 AZ는 서로 **100km 이내**
- ⚠️ 계정마다 AZ 코드와 실제 위치가 다름 (보안 + 쏠림 방지)

### 4-4. 엣지 로케이션(Edge Location) ★★

- **CDN(CloudFront)** 캐싱 거점 = 출장소 개념
- 리전/AZ보다 훨씬 많음 (전 세계 수백 개)
- 사용자와 가장 가까운 곳에서 콘텐츠 제공
- 예) 미국 넷플릭스를 한국에서 볼 때 → 한국 엣지 로케이션에서 캐시 제공

```
숫자 크기: 리전 (~30개) < 가용영역 (~100개) << 엣지 로케이션 (~600개+)
```

| 서비스 | 설명 |

### 6-5. 보안

| 서비스 | 설명 |
|--------|------|
| **IAM** | 사용자 및 권한 관리. **글로벌 서비스** |

## 📌 PART 8. EC2 상세

### 8-1. EC2 인스턴스 상태 (수명주기) ★★★

| 상태 | 설명 |
|------|------|
| **pending** | 대기 중 (시작 처리 중) |
| **running** | 실행 중 |
| **stopping** | 중지 중 |
| **stopped** | 중지됨 (재시작 가능, EBS 유지) |
| **shutting-down** | 종료 중 |
| **terminated** | 종료됨 (완전 삭제, 복구 불가) |
| **rebooting** | 재부팅 중 |

### 8-2. 중지(Stop) vs 종료(Terminate) ★★★

| | 중지 (Stop) | 종료 (Terminate) |
|--|------------|----------------|
| 의미 | 전원 OFF (재시작 가능) | **완전 삭제** |
| EBS 데이터 | **유지됨** | 삭제됨 (기본) |
| IP 주소 | 변경될 수 있음 | 없어짐 |
| 복구 | 가능 | **불가능** ⚠️ |

> **인스턴스 유형 변경은 반드시 중지 상태에서만 가능!**

### 8-3. PEM vs PPK ★★

| | .pem | .ppk |
|--|------|------|
| 풀네임 | Privacy-Enhanced Mail | PuTTY Private Key |
| 사용 OS | **Mac / Linux** | **Windows** |
| SSH 클라이언트 | OpenSSH (ssh 명령어) | PuTTY |
| 변환 | pem → ppk : **PuTTYgen** 사용 | |

### 8-4. EC2 접속 방법 (Windows 기준)

```
1. 퍼블릭 IP 주소 복사
2. putty.exe 실행
3. HostName에 IP 입력
4. SSH → Auth → Credentials → Browse → ppk 파일 선택
5. login as: ec2-user 입력
```

## 📌 PART 9. Security Group (보안 그룹)

- EC2 앞에 세워진 **가상 방화벽**
- **인바운드 규칙:** 외부 → EC2 (허용할 포트/프로토콜/소스 지정)
- **아웃바운드 규칙:** EC2 → 외부 (기본: 모든 트래픽 허용)

| 포트 | 프로토콜 | 용도 |
|------|---------|------|
| 22 | SSH | 터미널 접속 |
| 80 | HTTP | 웹서버 접속 |
| 443 | HTTPS | 보안 웹서버 |
| ICMP | - | ping (네트워크 연결 확인) |

> **웹페이지가 안 보인다면?** → 보안그룹에 HTTP(80) 인바운드 규칙 없는 것이 원인!

## 📌 PART 10. CloudWatch & SNS ★★★

### 10-1. CloudWatch = 클라우드 운영의 눈(Eye)

AWS 리소스를 실시간으로 **관찰·기록·대응**하는 모니터링 서비스

| 기능 | 설명 |
|------|------|
| **Metrics (지표)** | CPU, 메모리 등 숫자 데이터 |
| **Logs (로그)** | 텍스트 형태의 기록 |
| **Alarm (알람)** | 임계값 초과 시 알림 발송 → SNS 연동 |
| **Dashboard (대시보드)** | 모든 상태를 한 화면에서 확인 |

### 10-2. SNS = Simple Notification Service ★

- AWS의 **알림 발송 전용 서비스**
- 이메일, SMS, Lambda, SQS 등 다양한 채널로 알림 전달
- 구독(Subscribe) 후 **이메일로 Confirm subscription 승인 필수!**

### 10-3. CloudWatch + SNS 연동 흐름

```
CPU > 30% 초과
  → CloudWatch 경보 발동
    → SNS 주제(Topic)로 메시지 전송
      → 구독된 이메일로 알림 도착
```

### Git 비교 총정리

| 비교 | A | B |
|------|---|---|

| 비교 | A | B |
|------|---|---|
| **중지 vs 종료** | 재시작 가능, EBS 유지 | 완전 삭제, 복구 불가 |

| **CloudWatch vs SNS** | 모니터링/감지 | 알림 발송 |
| **S3 vs EBS** | 객체 스토리지(인터넷) | 블록 스토리지(EC2 전용) |

## 📝 예상 주관식 문제 & 답

**Q1. git reset의 3가지 옵션과 각각의 차이점을 설명하시오.**  
→ --soft: Staging Area로 이동 / --mixed: Working Directory로 이동(기본값) / --hard: 완전 삭제

**Q2. reset과 revert의 차이점을 설명하시오.**  
→ reset은 이후 커밋을 삭제하고 돌아감. revert는 되돌리는 새 커밋을 추가해 기록을 보존. 협업에서는 revert가 안전.

**Q3. merge와 rebase의 차이점을 설명하시오.**  
→ merge는 두 브랜치를 합쳐 새 커밋 생성(분기 형태). rebase는 커밋을 다른 브랜치 위에 재배치(히스토리 일직선).

**Q4. EC2 인스턴스의 중지와 종료의 차이점을 설명하시오.**  
→ 중지는 전원 OFF로 재시작 가능하고 EBS 데이터가 유지됨. 종료는 완전 삭제로 복구 불가.

**Q5. CloudWatch와 SNS의 역할을 설명하고 둘의 관계를 설명하시오.**  
→ CloudWatch는 AWS 리소스 모니터링 서비스(지표,로그,알람,대시보드). SNS는 알림 발송 서비스. CloudWatch 경보 발동 시 SNS를 통해 이메일 등으로 알림 전송.

**Q6. 글로벌 서비스와 리전 서비스의 차이점과 각 예시를 쓰시오.**  
→ 글로벌: 전 세계 공유, 리전 선택 불필요 (IAM, Route53, WAF, Billing). 리전: 특정 리전 기반 (EC2, S3, RDS, VPC, Lambda).

**Q9. VPC란 무엇인지 설명하시오.**  
→ Virtual Private Cloud. AWS 계정 전용 가상 네트워크. 외부와 격리된 사설망 구축. 리전 단위로 생성.

**Q10. 엣지 로케이션이란 무엇이고 어떤 서비스에 사용되는지 설명하시오.**  
→ CloudFront(CDN) 등의 콘텐츠를 사용자와 가장 가까운 곳에서 빠르게 캐싱해 제공하는 거점. 리전/AZ보다 훨씬 많고 전 세계에 분포.
