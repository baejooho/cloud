# PART 1. Git 초급
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 1-1. Git 3단계 영역 ★★★

```
Working Directory  →(git add)→  Staging Area  →(git commit)→  Repository
   내가 작업하는 곳              커밋 대기 공간                  영구 저장소
```

- **Working Directory**: 소스코드를 직접 작업하는 프로젝트 디렉토리
- **Staging Area**: git add 명령어로 추가한 파일들이 모여있는 공간
- **Repository**: commit 실행 시 최종적으로 Git 저장소에 반영되는 곳
- **Commit**: 변경 사항을 Git 저장소에 **영구적으로 기록**하는 작업

> 예) `git commit 파일 -m "로그메시지"`

---

## 1-2. 파일의 생명주기 4가지 ★★★

| 상태 | 설명 | 언제? |
|------|------|-------|
| **Untracked** | Git이 추적 안 함 | git add를 **한 번도** 안 한 파일 |
| **Unmodified** | 수정 없음 = new file | add 후 commit 완료, 수정사항 없음 |
| **Modified** | 수정된 상태 | commit 이후 파일을 수정했을 때 |
| **Staged** | Staging Area에 올라간 상태 | git add 실행 후 |

**실습으로 확인:**
```bash
git status        # → Untracked: a.txt, b.txt 표시
git add a.txt     # → Unmodified (new file: a.txt)
# a.txt 파일 수정
git status        # → Modified: a.txt
git commit a.txt -m "파일생명주기를 위해 a.txt추가"
```

---

## 1-3. reset 3가지 옵션 ★★★ (주관식 단골!)

| 옵션 | 파일 이동 위치 | 설명 |
|------|--------------|------|
| `--soft` | Staging Area | commit 직전 상태. 파일은 staged 유지 |
| `--mixed` | Working Directory | add 직전 상태 **(기본값, default)** |
| `--hard` | 완전 삭제 | tracked 파일을 working directory에서 삭제. 복구 불가 ⚠️ |

```bash
git reset --soft [commit ID]
git reset --mixed [commit ID]   # = git reset [commit ID] (기본값)
git reset --hard [commit ID]
```

**HEAD란?** 현재 작업 중인 브랜치나 커밋을 가리키는 포인터

**reset 시각화:**
```
Before: A → B → C → D (HEAD)
--soft/mixed 로 B로 reset: A → B (HEAD), C·D 제거, 파일은 남음
--hard 로 B로 reset:       A → B (HEAD), C·D 제거, 파일도 삭제
```

---

## 1-4. reset vs revert ★★★

| 구분 | reset | revert |
|------|-------|--------|
| 방식 | 원하는 시점으로 돌아가고 **이후 커밋 삭제** | 되돌리는 **새 커밋을 추가** |
| 기록 | 삭제됨 | **보존됨** |
| 협업 | ⚠️ 위험 | ✅ **안전** (충돌 최소화) |
| 장점 | - | 왜 돌아갔는지 기록 가능 |

```bash
git reset --hard [commit ID]    # 해당 시점으로 돌아가고 이후 삭제
git revert [commit ID]          # 되돌리는 새 커밋 생성
```

**revert 시각화:**
```
Before:  A → B → C → D (HEAD)
After revert C:
         A → B → C → D → D' → C' (HEAD)
                         역순으로 새 커밋이 쌓임
```

**SourceTree에서:**
- reset: 커밋 우클릭 → "이 커밋까지 현재 브랜치를 초기화" → Soft/Mixed/Hard 선택
- revert: 커밋 우클릭 → "커밋 되돌리기"

---

## 1-5. 기본 명령어

```bash
git init                         # 저장소 초기화
git status                       # 현재 상태 확인
git add <파일> / git add .        # Staging Area로 이동
git commit -m "메시지"           # 커밋
git log                          # 커밋 내역 확인
git log --all --oneline --graph  # 그래프 형태로 전체 확인

# 계정 초기화
git config --global --unset user.name
git config --global --unset user.email
git config --global --list       # 설정 확인
```

# PART 2. Git 브랜치 (중급)
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 2-1. 브랜치(Branch)란? ★★

- 특정 시점에서 코드의 버전(상태)을 나누어 **별도로 관리**하는 기능
- 기본 브랜치(Main)에서 **나뭇가지처럼** 새로운 브랜치 생성
- 기능 추가, 오류 수정 등을 **독립적으로** 작업 가능

---

## 2-2. 브랜치 명령어 6개 ★★★

```bash
git branch                          # 브랜치 목록 확인
git branch <브랜치명>               # 브랜치 생성
git switch <브랜치명>               # 브랜치 이동
git switch -c <브랜치명>            # 생성 + 이동 동시에 ★
git branch -m <기존이름> <새이름>   # 브랜치 이름 수정
git branch -d <브랜치명>            # 브랜치 삭제
```

**전체 그래프 확인:**
```bash
git log --all --decorate --oneline --graph
```

---

## 2-3. merge vs rebase ★★★

| 구분 | merge | rebase |
|------|-------|--------|
| 의미 | 두 브랜치 합쳐 **새 커밋 생성** | 커밋을 다른 브랜치 위에 **재배치** |
| 그래프 | 분기 형태 (갈라진 모양) | **히스토리 일직선** |
| 명령어 | `git merge <브랜치명>` | `git rebase <브랜치명>` |
| 취소 | `git merge --abort` | `git rebase --abort` |

**merge 흐름:**
```bash
git switch main         # main으로 이동
git merge leader        # leader를 main에 병합
git log                 # 커밋 내역 확인
git branch -d leader    # 병합된 브랜치 삭제
```

**rebase 흐름:**
```bash
git switch manager      # manager로 이동
git rebase main         # manager 커밋들을 main 위에 재배치
# → main 브랜치가 뒤쳐져 있음! 반드시 아래 실행
git switch main
git merge manager       # main을 manager 시점으로 이동 (fast-forward)
git branch -d manager   # 브랜치 삭제
```

> ⚠️ **rebase 핵심 포인트:** rebase 후에는 main이 뒤처지므로 반드시 `git merge`로 당겨줘야 함!

---

## 2-4. 충돌(Conflict) 해결 ★★

**충돌 표시 읽는 법:**
```
<<<<<<< HEAD          ← 현재 브랜치(main)의 내용
현재 내용
=======
상대방 내용
>>>>>>> 브랜치명      ← 합치려는 브랜치 내용
```

| 구분 | merge 충돌 | rebase 충돌 |
|------|-----------|-------------|
| 취소 | `git merge --abort` | `git rebase --abort` |
| 해결 순서 | 파일수정 → `git add` → `git commit` | 파일수정 → `git add` → `git rebase --continue` |

---

## 2-5. merge 후 원복

```bash
# 병합 이전 커밋으로 reset (hard)
git reset --hard [병합 전 commit ID]
git branch -d leader    # 브랜치 삭제
```

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 3-1. push / pull 개념

- **GitHub**: 원격 저장소 (클라우드에 있는 Git 저장소)
- **git push**: 로컬 → 원격. 변경사항(커밋)을 원격 저장소로 업로드
- **git pull**: 원격 → 로컬. 원격 저장소의 변경사항을 로컬로 가져옴

```
로컬 저장소  →(git push)→  GitHub(원격)
로컬 저장소  ←(git pull)←  GitHub(원격)
```

---

## 3-2. 최초 push 3단계 순서 ★★★

```bash
# 1단계: 원격 저장소 연결
git remote add origin https://github.com/계정/레포.git

# 2단계: 브랜치 이름 main으로 설정
git branch -M main

# 3단계: 최초 push (-u로 upstream 설정)
git push -u origin main
```

> 이후 push는 `git push` 만 해도 됨!

---

## 3-3. push 전 pull 필수! ★★★

```bash
git pull origin main    # ← 반드시 먼저!
git add .
git commit -m "메시지"
git push origin main    # ← 나중에!
```

> 이유: 원격 저장소의 최신 변경사항을 먼저 반영해야 충돌 예방 가능!

---

## 3-4. clone vs pull

| 구분 | git clone | git pull |
|------|-----------|----------|
| 사용 시점 | **최초 1회**, 전체 복사 | 이미 있는 로컬에 **변경사항만** |
| 명령어 | `git clone <URL>` | `git pull origin main` |

---

## 3-5. Access Token ★★

- GitHub는 **2021년 8월부터** 비밀번호 인증 제거
- push 시 password 자리에 **Access Token** 입력 필요
- 오류: `Authentication failed` → **Access Token 미설정이 원인**

---

## 3-6. 다른 브랜치에 push

```bash
git switch leader
git push -u origin leader
```

---

## 3-7. 협업 흐름

1. 초대 보내는 사람: GitHub 레포 → Settings → Collaborators → 초대
2. 초대 받는 사람: 이메일에서 수락(Accept)
3. `git clone <URL>` → 작업 → `git pull` → `git add` → `git commit` → `git push`

# PART 4. AWS 핵심 구조
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 4-1. AWS 계층 구조 ★★★

```
리전 (Region)              ← 가장 큰 단위
  └── 가용영역 (AZ)        ← 리전 하부
        └── VPC            ← 리전 단위
              └── 서브넷   ← AZ 단위
                    └── EC2 인스턴스
```

## 4-2. 리전 핵심 암기

| 리전 | 코드 | 특징 |
|------|------|------|
| 서울 | `ap-northeast-2` | 한국 서비스 기본 리전 |
| **버지니아 북부** | **`us-east-1`** | **★ 모든 AWS 서비스 최초 출시** |

**리전 선택 기준:** 지연속도 / 법률(데이터 주권) / 서비스 가용성

## 4-3. 가용영역(AZ)

- 리전 안에 **2개 이상** 존재
- 물리 데이터센터를 묶은 **논리적** 데이터센터
- 서로 몇 km 이상 떨어짐 (재해 대비), 단 **100km 이내**
- ⚠️ 계정마다 AZ 코드와 실제 위치가 다름 (보안 + 쏠림 방지)

## 4-4. 엣지 로케이션 ★★

- CloudFront(CDN) 콘텐츠 **캐싱 거점** = 출장소 개념
- 숫자: **리전(~30) < AZ(~100) << 엣지 로케이션(~600+)**
- 예) 미국 넷플릭스 → 한국 엣지 로케이션에서 캐시 제공 → 빠름!

## 4-5. 글로벌 vs 리전 서비스 ★★★

| 구분 | 글로벌 서비스 | 리전 서비스 |
|------|--------------|------------|
| 개념 | 전 세계 인프라 공유 | 특정 리전 기반 |
| **예시** | **IAM, Route53, WAF, Billing** | **EC2, S3, RDS, VPC, Lambda** |

> 💡 이 4개만 글로벌! (IAM·Route53·WAF·Billing) 나머지 전부 리전!

# PART 5. S3 버킷 정책 ★★★ (신규 출제 예상!)
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 5-1. S3 기본 개념

- **객체 스토리지**: 파일을 인터넷으로 저장하고 접근
- **정적 웹 호스팅** 가능 (HTML, CSS, JS, 이미지 등)
- 기본적으로 모든 객체는 **Private(비공개)** 상태
- 퍼블릭 접근은 **명시적으로 허용**해야 함

---

## 5-2. IAM 정책 2종류 ★★★

| 종류 | 부여 대상 | 핵심 질문 |
|------|----------|----------|
| **Identity-based Policy** (자격 증명 기반) | IAM 사용자·그룹·역할 | "**누가** 무엇을 할 수 있는가" |
| **Resource-based Policy** (리소스 기반) | S3·SQS 등 리소스에 직접 | "이 리소스에 **누가** 접근 가능한가" |

---

## 5-3. S3 버킷 정책 JSON 구조 ★★★

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "정책이름",
      "Effect": "Allow 또는 Deny",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::버킷명/*"
    }
  ]
}
```

**각 항목 설명:**

| 항목 | 설명 |
|------|------|
| `Version` | 정책 버전 (항상 "2012-10-17") |
| `Sid` | 정책 식별자(이름) |
| `Effect` | Allow(허용) 또는 Deny(거부) |
| `Principal` | 접근 주체 (`*` = 모든 사람) |
| `Action` | 허용/거부할 행동 (s3:GetObject = 다운로드) |
| `Resource` | 적용할 리소스 ARN |

---

## 5-4. 버킷 정책 케이스별 완전 정리 ★★★

### 케이스 1: 버킷 전체 퍼블릭 허용

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicWebsiteAllow",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::sgu-202500-s3/*"
  }]
}
```
> `/*` = 버킷 안의 **모든 파일**에 적용

---

### 케이스 2: 특정 파일/폴더만 허용 ★

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicWebsiteAllow",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": [
      "arn:aws:s3:::sgu-202500-s3-test/index.html",
      "arn:aws:s3:::sgu-202500-s3-test/class/img/*"
    ]
  }]
}
```
> Resource를 **배열[]** 로 지정 → 여러 경로 설정 가능

---

### 케이스 3: 특정 IP만 접근 허용 ★★

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "PublicWebsiteAllow",
    "Effect": "Allow",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": [
      "arn:aws:s3:::sgu-202500-s3-test/index.html",
      "arn:aws:s3:::sgu-202500-s3-test/class/img/*"
    ],
    "Condition": {
      "IpAddress": {
        "aws:SourceIp": "123.123.123.123"
      }
    }
  }]
}
```
> **Condition + IpAddress + aws:SourceIp** 로 IP 제한

---

### 케이스 4: 특정 폴더 다운로드 차단 (Deny) ★★

```json
{
  "Version": "2012-10-17",
  "Statement": [{
    "Sid": "DenyDownload",
    "Effect": "Deny",
    "Principal": "*",
    "Action": "s3:GetObject",
    "Resource": "arn:aws:s3:::버킷명/class/test/*"
  }]
}
```
> `Effect: "Deny"` + `Action: "s3:GetObject"` = **다운로드 차단**
> 업로드는 허용, 다운로드만 막을 때 사용

---

## 5-5. 정책 핵심 암기표

| 목적 | Effect | Action | 추가 설정 |
|------|--------|--------|----------|
| 전체 공개 | Allow | s3:GetObject | Resource: `버킷/*` |
| 특정 파일만 공개 | Allow | s3:GetObject | Resource: 배열[] |
| 특정 IP만 허용 | Allow | s3:GetObject | Condition + IpAddress |
| 다운로드 차단 | **Deny** | **s3:GetObject** | Resource: 제한 경로 |

## 6-1. 중지(Stop) vs 종료(Terminate) ★★★

| 구분 | 중지 (Stop) | 종료 (Terminate) |
|------|------------|----------------|
| 의미 | 전원 OFF. 재시작 가능 | **완전 삭제** |
| EBS | **유지됨** | 삭제됨 (기본) |
| IP | 변경될 수 있음 | 없어짐 |
| 복구 | 가능 | **불가능** ⚠️ |

> 💡 인스턴스 **유형 변경은 중지 상태**에서만 가능!

## 6-2. 인스턴스 수명주기 7가지

| 상태 | 설명 |
|------|------|
| pending | 대기 중 |
| **running** | 실행 중 |
| stopping | 중지 중 |
| **stopped** | 중지됨 (EBS 유지, 재시작 가능) |
| shutting-down | 종료 중 |
| **terminated** | 종료됨 (완전 삭제, 복구 불가) |
| rebooting | 재부팅 중 |

## 6-3. PEM vs PPK ★★

| 구분 | .pem | .ppk |
|------|------|------|
| 풀네임 | Privacy-Enhanced Mail | PuTTY Private Key |
| OS | **Mac/Linux** | **Windows** |
| 변환 | pem→ppk: **PuTTYgen** | - |

## 6-4. Security Group 주요 포트

| 포트 | 프로토콜 | 용도 |
|------|---------|------|
| 22 | SSH | 터미널 접속 |
| 80 | HTTP | 웹서버 |
| 443 | HTTPS | 보안 웹서버 |
| ICMP | - | ping |

> 💡 웹페이지 안 보이면? → 보안그룹 **HTTP(80) 인바운드** 규칙 확인!

# PART 7. CloudWatch & SNS & IAM
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

## 7-1. CloudWatch = 클라우드 운영의 눈 ★★★

| 기능 | 설명 |
|------|------|
| **Metrics (지표)** | CPU, 메모리 등 숫자 데이터 |
| **Logs (로그)** | 텍스트 형태의 기록 |
| **Alarm (알람)** | 임계값 초과 → SNS 연동 알림 |
| **Dashboard** | 모든 상태를 한 화면에서 확인 |

## 7-2. SNS = Simple Notification Service ★★

- AWS의 **알림 발송 전용 서비스**
- 이메일, SMS 등으로 알림 전달
- 구독 후 **이메일에서 Confirm subscription 승인 필수!**

**연동 흐름:**
```
CPU 30% 초과 → CloudWatch 경보 → SNS 주제 → 이메일 알림
```

## 7-3. IAM 핵심 ★★★

| 요소 | 설명 |
|------|------|
| **User** | 개별 사용자 계정 |
| **Group** | 여러 사용자 묶어 동일 권한 부여 |
| **Role** | 임시 권한 부여 (AWS 서비스 간 위임) |
| **Policy** | JSON 형식의 권한 규칙 |

**보안 주의사항:**
- 루트 계정 최소 사용 + **MFA 필수**
- **최소 권한 원칙** (꼭 필요한 것만)
- 액세스 키 **GitHub에 절대 올리지 않기** → 해킹 시 수억 원 청구 가능!
- IAM은 **글로벌 서비스**

# PART 8. 전체 명령어 모음
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

```bash
# 기본
git init / git status / git add . / git commit -m ""
git log / git log --all --oneline --graph

# 되돌리기
git reset --soft [ID]       # Staging으로
git reset --mixed [ID]      # Working Dir로 (기본값)
git reset --hard [ID]       # 완전 삭제
git revert [ID]             # 새 커밋으로 되돌리기

# 브랜치
git branch / git branch <이름> / git switch <이름>
git switch -c <이름>        # 생성+이동 동시에 ★
git branch -m <기존> <새>
git branch -d <이름>
git merge <브랜치명>
git merge --abort           # merge 취소
git rebase <브랜치명>
git rebase --abort          # rebase 취소
git rebase --continue       # 충돌 해결 후 계속

# GitHub
git remote add origin <URL>
git branch -M main
git push -u origin main     # 최초 push
git push                    # 이후 push
git pull origin main
git clone <URL>

# 계정 초기화
git config --global --unset user.name
git config --global --unset user.email
git config --global --list
```

**[Git 초급 - 약 8문제 예상]**

**Q1. Git의 3단계 영역을 순서대로 쓰고 각각 설명하시오.**
> Working Directory(작업 공간) → Staging Area(커밋 대기 공간) → Repository(영구 저장소)

**Q2. 파일의 생명주기 4가지를 쓰고 각각 설명하시오.**
> Untracked(한 번도 add 안 함) / Unmodified(commit 완료, 수정 없음=new file) / Modified(수정됨) / Staged(add 후 대기 중)

**Q3. git reset의 3가지 옵션과 차이점을 설명하시오.**
> --soft: Staging Area로 이동 / --mixed: Working Directory로 이동(기본값) / --hard: 완전 삭제

**Q4. reset과 revert의 차이점을 설명하시오.**
> reset은 이후 커밋을 삭제하고 돌아감. revert는 되돌리는 새 커밋을 추가해 기록 보존. 협업에서는 revert가 안전.

**Q5. HEAD란 무엇인지 설명하시오.**
> 현재 작업 중인 브랜치나 커밋을 가리키는 포인터

**Q6. git commit이란 무엇인가?**
> 변경 사항을 Git 저장소에 영구적으로 기록하는 작업. Staging Area에서 Repository로 이동.

**Q7. git add를 하면 파일이 어디로 이동하는가?**
> Staging Area

**Q8. git add를 한 번도 안 한 파일의 상태는?**
> Untracked

---

**[브랜치 - 약 6문제 예상]**

**Q9. 브랜치(Branch)란 무엇인지 설명하시오.**
> 특정 시점에서 코드의 버전을 나누어 별도로 관리하는 기능. 기본 브랜치에서 나뭇가지처럼 새 브랜치를 생성하여 독립적 작업 가능.

**Q10. 브랜치 생성과 이동을 동시에 하는 명령어는?**
> `git switch -c <브랜치명>`

**Q11. merge와 rebase의 차이점을 설명하시오.**
> merge: 두 브랜치 합쳐 새 커밋 생성(분기 형태). rebase: 커밋을 다른 브랜치 위에 재배치(히스토리 일직선).

**Q12. merge 충돌 해결 순서를 쓰시오.**
> 파일 수정 → git add → git commit

**Q13. rebase 충돌 해결 순서를 쓰시오.**
> 파일 수정 → git add → git rebase --continue

**Q14. 충돌 표시에서 HEAD는 무엇을 의미하는가?**
> 현재 작업 중인 브랜치(main)의 내용

**[GitHub 고급 - 약 4문제 예상]**

**Q15. 로컬을 GitHub에 최초 push하는 3단계 명령어를 순서대로 쓰시오.**
> ① git remote add origin \<URL\> → ② git branch -M main → ③ git push -u origin main

**Q16. push 전에 pull을 먼저 해야 하는 이유는?**
> 원격 저장소의 최신 변경사항을 먼저 반영하여 충돌을 예방하기 위해.

**Q17. Authentication failed 오류의 원인과 해결 방법은?**
> 2021년 8월부터 GitHub 비밀번호 인증 제거. password 자리에 Access Token 입력 필요.

**Q18. git clone과 git pull의 차이점은?**
> clone: 최초 1회 전체 복사. pull: 이미 있는 로컬에 변경사항만 업데이트.

**[AWS 구조 - 약 4문제 예상]**

**Q19. AWS의 계층 구조를 큰 것부터 순서대로 쓰시오.**
> 리전 → 가용영역(AZ) → VPC → 서브넷 → EC2 인스턴스

**Q20. us-east-1 리전의 특징은?**
> 미국 버지니아 북부 리전. 모든 AWS 서비스가 최초로 출시되는 리전.

**Q21. 글로벌 서비스와 리전 서비스의 차이점과 예시를 쓰시오.**
> 글로벌: 전 세계 공유, 리전 선택 불필요 (IAM, Route53, WAF, Billing). 리전: 특정 리전 기반 (EC2, S3, RDS, VPC, Lambda).

**Q22. 엣지 로케이션이란 무엇이고 어떤 서비스에 사용되는지 설명하시오.**
> CloudFront(CDN)의 콘텐츠를 사용자와 가장 가까운 곳에서 캐싱해 빠르게 제공하는 거점. 리전/AZ보다 훨씬 많음.

**[S3 & EC2 & IAM - 약 3문제 예상]**

**Q23. S3 버킷 정책에서 Effect: Deny와 Action: s3:GetObject를 함께 사용하면?**
> 해당 리소스의 다운로드(읽기)를 차단. 업로드는 허용, 다운로드 불가.

**Q24. S3 버킷 정책에서 특정 IP만 접근 허용하려면?**
> Condition 블록에 IpAddress와 aws:SourceIp를 추가하여 IP 조건 설정.

**Q25. EC2 중지와 종료의 차이점, 인스턴스 유형 변경 시 필요한 상태는?**
> 중지: 전원 OFF, 재시작 가능, EBS 유지. 종료: 완전 삭제, 복구 불가. 유형 변경은 반드시 **중지 상태**에서만 가능.

# ⚡ 최종 비교 요약표 (시험 직전 확인!)

| 비교 | A | B |
|------|---|---|
| reset vs revert | 커밋 삭제 (위험) | 새 커밋 추가 (안전) |
| merge vs rebase | 분기 형태 새 커밋 | 일직선 재배치 |
| clone vs pull | 최초 전체 복사 | 변경사항만 |
| --soft vs --hard | Staging으로 | 완전 삭제 |
| 중지 vs 종료 | 재시작 가능·EBS유지 | 완전삭제·복구불가 |
| pem vs ppk | Mac/Linux | Windows/PuTTY |
| 글로벌 vs 리전 | IAM·Route53·WAF·Billing | EC2·S3·RDS·VPC·Lambda |
| CloudWatch vs SNS | 모니터링·감지 | 알림 발송 |
| S3 vs EBS | 객체 스토리지(인터넷) | 블록 스토리지(EC2 전용) |
| Allow vs Deny | 허용 | 차단 **(Deny 우선!)** |
| Identity-based vs Resource-based | 사용자 기준 | 리소스 기준 |
