# ELB (Elastic Load Balancer)

## 1. ELB란? 

AWS에서 제공하는 **트래픽 분산 서비스**.

ELB가 없다면 사용자가 EC2에 접근하려면 모든 EC2의 IP 주소를 직접 알아야 한다. EC2가 수십~수백 대로 늘어나면 사실상 관리가 불가능해진다. 이 문제를 해결하기 위해 ELB를 사용한다.

> **비유**: ELB는 대형 콜센터의 교환원이다. 고객(사용자)은 대표번호(ELB 도메인) 하나만 알면 되고, 교환원이 알아서 한가한 상담원(EC2)에게 연결해준다.

---

## 2. ELB 특징

| 특징 | 설명 |
|---|---|
| **트래픽 분산** | 다수의 EC2 인스턴스에 트래픽을 자동으로 나눠준다 |
| **Health Check** | 직접 트래픽을 발생시켜 인스턴스가 살아있는지 주기적으로 체크한다 |
| **Auto Scaling 연동** | 트래픽에 따라 EC2를 자동으로 늘리고 줄이는 Auto Scaling과 연동 가능하다 |
| **다중 가용영역** | 여러 AZ(가용영역)에 분산 배치가 가능하다 |
| **IP 고정 불가** | IP 주소가 지속적으로 바뀌므로, 항상 **도메인 기반**으로 접근해야 한다 |

- ELB에 접근할 때는 아래처럼 도메인을 사용한다:
  ```
  test.ap-northeast-2.elb.amazonaws.com
  ```

## 3. ELB 종류 (3가지)

### 3-1. Application Load Balancer (ALB) — 똑똑함 ⭐

- **HTTP/HTTPS** 트래픽을 처리한다
- **URL 경로, 호스트 헤더, 쿼리 스트링** 등을 기반으로 라우팅 가능하다
- 예시:
  - `image.test.com` → 이미지 서버로 전달
  - `web.test.com` → 웹 서버로 전달
  - `/api` 경로 → API 서버로 전달
- 마이크로서비스, 컨테이너 기반 아키텍처에 적합하다

### 3-2. Network Load Balancer (NLB) — 빠름

- **TCP 기반** 빠른 트래픽 분산을 처리한다
- **Elastic IP 할당 가능** (IP 고정 가능)
- 초저지연, 대용량 트래픽 처리에 적합하다

### 3-3. Classic Load Balancer (CLB) — 레거시

- 예전 방식이며 현재는 거의 사용하지 않는다

## 4. ALB - Target Group (대상 그룹)

**타겟 그룹**이란 ALB가 트래픽을 분산시킬 대상(Target)을 논리적으로 묶어놓은 그룹이다.

### 타겟으로 등록 가능한 대상
- EC2 인스턴스
- IP 주소
- Lambda 함수
- 다른 ELB

### 동작 방식
```
사용자 요청
    ↓
ALB (라우팅 규칙 확인)
    ↓
Target Group
    ↓
Health Check 통과한 인스턴스에만 트래픽 전달
```

### 타겟 그룹 설정 항목
- 포트, 프로토콜, 헬스체크 설정을 타겟 그룹별로 다르게 지정 가능하다

### ALB 구성 예시
```
www.example.com
    ├── /orders       → Orders EC2 Target Group
    ├── /images       → Images EC2 Target Group
    └── /registration → Registration EC2 Target Group

각 Target Group은 AZ별 Auto Scaling Group으로 구성됨
```

## 5. 실습 1 — ALB 생성

### 전체 순서
1. 시작 템플릿 수정
2. 시작 템플릿으로 EC2 2대 생성
3. Target Group 생성 (대상 그룹)
4. ALB 생성

### 시작 템플릿 User Data 스크립트
EC2가 시작될 때 자동으로 실행되는 스크립트. 인스턴스 ID를 웹페이지로 출력한다.

```bash
#!/bin/bash
yum install httpd -y
systemctl enable httpd
systemctl start httpd

# IMDSv2 방식으로 토큰 발급
TOKEN=$(curl -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# 인스턴스 ID 가져오기
INSTANCE_ID=$(curl -H "X-aws-ec2-metadata-token: $TOKEN" \
  -s http://169.254.169.254/latest/meta-data/instance-id)

# 웹 페이지에 인스턴스 ID 출력
echo "$INSTANCE_ID" > /var/www/html/index.html
```

> 브라우저로 ELB 도메인에 접근하면 매 요청마다 다른 인스턴스 ID가 표시된다 → 트래픽 분산 확인 가능

### Target Group 생성 설정
- **대상 유형**: 인스턴스
- **프로토콜**: HTTP / 포트: 80
- **VPC**: 사용 중인 VPC 선택
- EC2 인스턴스 2대를 등록 후 `아래에 보류 중인 것으로 포함` 클릭

### ALB 생성 설정
- **이름**: `{계정}-ALB`
- **체계**: 인터넷 경계 (Internet-facing)
- **가용영역**: 전체 선택
- **보안그룹**: 사용 중인 보안그룹 선택
- **리스너**: HTTP:80 → 생성한 Target Group으로 전달
- 생성 후 `프로비저닝 중` 상태를 거쳐 `활성` 상태가 된다

## 6. 실습 2 — 경로 기반 리디렉션 (ALB → S3)

`/s3` 경로로 들어온 요청을 S3 정적 웹사이트로 리디렉션하는 실습.

### 전체 순서
1. S3 버킷에 정적 웹사이트 호스팅 활성화
2. ALB 리스너에 규칙 추가

### S3 설정
- `버킷 → 속성 → 정적 웹 사이트 호스팅 → 활성화`
- 인덱스 문서: `index.html`
- 활성화 후 **버킷 웹 사이트 엔드포인트** 확인

  ```
  http://{버킷이름}.s3-website.ap-northeast-2.amazonaws.com
  ```

### ALB 리스너 규칙 추가
`해당 ALB → 리스너 및 규칙 → 규칙 관리 → 규칙 추가`

| 설정 항목 | 값 |
|---|---|
| 규칙 이름 | `redirection to S3` |
| 조건 (규칙 조건) | 경로 패턴: `/s3` |
| 작업 유형 | URL로 리디렉션 |
| 리디렉션 대상 | S3 버킷 웹사이트 엔드포인트 + 경로 `/s3` |
| 상태 코드 | `302 - 찾음` (임시 리디렉션) |
| 우선 순위 | `1` (가장 먼저 평가됨) |

### 최종 리스너 규칙 구조
```
HTTP:80 리스너
  ├── [우선순위 1] 경로 패턴 = /s3  → S3 버킷으로 302 리디렉션
  └── [기본값] 그 외 모든 요청     → Target Group으로 전달
```

### 확인
```
ALB도메인/s3 접근 시
    → ALB가 302 응답 반환
    → 브라우저가 S3 버킷 엔드포인트로 이동
    → S3의 index.html 페이지 표시
```

## 7. 핵심 요약

| 구분 | 내용 |
|---|---|
| ELB 역할 | 트래픽을 여러 EC2에 자동 분산 |
| IP 방식 | 도메인 기반 접근 (IP 고정 불가) |
| Health Check | 죽은 인스턴스엔 트래픽 안 보냄 |
| ALB 강점 | URL 경로/호스트 기반 스마트 라우팅 |
| Target Group | ALB와 EC2 사이의 논리적 그룹 |
| 경로 기반 라우팅 | /경로에 따라 서로 다른 서버로 분기 가능 |
