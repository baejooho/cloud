## 1. 보안 그룹 생성 (Security Group)

보안 그룹은 EC2 인스턴스에 대한 트래픽을 제어하는 방화벽 규칙 세트입니다.

### 생성 경로

```
AWS 콘솔 → EC2 → 네트워크 및 보안 → 보안 그룹 → 보안 그룹 생성
```

### 설정 예시

| 항목 | 값 |
|------|-----|
| 보안 그룹 이름 | `sgu-202601-EC2` |
| 설명 | `sgu-202601 EC2 Security Group` |
| VPC | `vpc-xxxxxxxx (shingu_class)` |

### 인바운드 규칙

| 유형 | 프로토콜 | 포트 범위 | 소스 |
|------|----------|-----------|------|
| SSH | TCP | 22 | 0.0.0.0/0 |

### 아웃바운드 규칙

| 유형 | 프로토콜 | 포트 범위 | 대상 |
|------|----------|-----------|------|
| 모든 트래픽 | 전체 | 전체 | 0.0.0.0/0 |

## 2. EC2 인스턴스 생성

### 접근 경로

```
AWS 콘솔 로그인 → EC2 서비스 검색 → 서울 리전 선택
→ 인스턴스 → 인스턴스 시작
```

### 설정 항목

| 항목 | 값 |
|------|-----|
| 이름 | `계정-서비스-test` (예: `sgu-202601-ec2test`) |
| AMI | Amazon Linux 2023 kernel-6.1 |
| 인스턴스 유형 | `t3.nano` |
| 키페어 | 새 키페어 생성 |

> ✅ 모든 설정 완료 후 **인스턴스 시작** 버튼 클릭

## 3. 키페어 생성 및 변환

### 키페어란?

| 형식 | 설명 | 사용 환경 |
|------|------|----------|
| `.pem` | Privacy-Enhanced Mail 형식 | Linux, macOS (ssh 명령어) |
| `.ppk` | PuTTY Private Key 형식 | Windows (PuTTY 클라이언트) |

### 키페어 생성

- **키페어 이름**: 임의 지정 (예: `anhr`, `sgu-202500`)
- **키페어 유형**: RSA
- **파일 형식**:
  - Windows → `.ppk`
  - macOS / Linux → `.pem`

> ⚠️ **중요**: 다운로드 받은 ppk/pem 파일은 반드시 **개인 메일로 백업**하세요! 재발급이 불가능합니다.

### PEM → PPK 변환 (Windows)

1. [https://www.putty.org](https://www.putty.org) 접속
2. `putty.zip` 다운로드 후 압축 해제
3. `puttygen.exe` 실행
4. `Load` 클릭 → 파일 형식을 `All Files (*.*)` 로 변경 → `.pem` 파일 선택 → 열기
5. 확인 메시지에서 `확인` 클릭
6. `Save private key` 클릭 → 경고창에서 `예(Y)` 클릭 → `.ppk` 파일 저장

## 4. EC2 접속

### 퍼블릭 IP 확인

```
EC2 콘솔 → 인스턴스 선택 → 인스턴스 요약 → 퍼블릭 IPv4 주소 복사
```

### PuTTY로 접속 (Windows)

1. `putty.exe` 실행
2. `Session` → **Host Name** 에 퍼블릭 IP 입력
3. `Connection → SSH → Auth → Credentials` → **Browse** → `.ppk` 파일 선택
4. `Open` 클릭
5. 보안 경고창 → `Accept`
6. 로그인 프롬프트에서 입력:

```
login as: ec2-user
```

### 접속 성공 화면

```
[ec2-user@ip-172-31-4-236 ~]$
```

## 6. 인스턴스 상태 및 수명 주기

### 인스턴스 상태 종류

| 상태 | 설명 |
|------|------|
| `pending` | 대기 중 (시작 처리 중) |
| `running` | 실행 중 |
| `stopping` | 중지 중 |
| `stopped` | 중지됨 |
| `shutting-down` | 종료 중 |
| `terminated` | 종료됨 (삭제 완료) |
| `rebooting` | 재부팅 중 |

### 수명 주기 흐름

```
최초 시작 → [대기 중(pending)] → [실행 중(running)]
                                        ↕ 재부팅
                              → [중지 중(stopping)] → [중지됨(stopped)]
                                                              ↓ 종료
                              → [종료 중(shutting-down)] → [종료됨(terminated)]
```

## 7. 인스턴스 유형 변경

> ⚠️ 인스턴스 유형 변경은 **반드시 중지 상태에서** 수행해야 합니다.

### 변경 경로

```
인스턴스 선택 → 작업 → 인스턴스 설정 → 인스턴스 유형 변경
```

### 실습: t3.nano → t3.micro 변경

1. 인스턴스 **중지**
2. `작업 → 인스턴스 설정 → 인스턴스 유형 변경` 클릭
3. `t3.micro` 선택 후 저장
4. 인스턴스 **시작**

## 8. 웹 서버 구축 실습

EC2에 Apache 웹 서버를 설치하고 간단한 웹 페이지를 구성합니다.

### 명령어 순서

```bash
# 슈퍼 유저로 전환
sudo su -

# Apache(httpd) 설치
yum install httpd -y

# Apache 시작
systemctl start httpd

# 웹 루트 디렉토리 이동
cd /var/www/html/

# index.html 생성
vi index.html
```

### index.html 내용

```html
<meta charset="utf-8">
<h1>Hello EC2 instance</h1>
<h3>웹서버 구축 성공!</h3>
```

### 웹페이지 확인

브라우저 주소창에 EC2 퍼블릭 IP 입력:

```
http://<퍼블릭IP>
```

> ❓ 페이지가 안 보인다면? → Security Group에 **HTTP(80포트) 인바운드 규칙**이 추가되어 있는지 확인!

## 9. CloudWatch 모니터링

### CloudWatch란?

AWS 리소스를 실시간으로 **관찰·기록·대응**하는 모니터링 서비스

| 기능 | 설명 |
|------|------|
| **Metrics (지표)** | 숫자로 보는 상태 데이터 (CPU, 네트워크 등) |
| **Logs (로그)** | 텍스트 형태의 기록 |
| **Alarm (알람)** | 임계값 초과 시 SNS 연동 알림 |
| **Dashboard (대시보드)** | 모든 상태를 한 화면에서 확인 |

### 9-1. 세부 모니터링 활성화

```
인스턴스 선택 → 모니터링 탭 → 세부 모니터링 관리 → 활성화 체크 → 확인
```

> 활성화 시 **1분 단위** 모니터링 데이터 수집 (기본은 5분)

### 9-2. CPU 사용률 대시보드 추가

```
모니터링 탭 → CPU 사용률 확장 → 지표에서 보기 클릭
→ 작업 → 대시보드에 추가 → 새로 생성
→ 대시보드 이름 입력 (예: sgu-202601-CloudWatch)
→ 생성 → 대시보드 추가 → 대시보드에 저장
```

### 9-3. CloudWatch 경보 생성 (High CPU Alarm)

```
CloudWatch → 경보 → 경보 생성 → 지표 선택
```

**지표 설정:**
```
검색: CPUUtilization → 인스턴스별 지표 → 내 인스턴스 체크
옵션 탭 → 누적 면적 → 지표 선택
```

**조건 설정:**

| 항목 | 값 |
|------|-----|
| 기간 | 1분 |
| 임계값 유형 | 정적 |
| 조건 | 보다 큼 (>) |
| 임계값 | `30` |
| 누락 데이터 처리 | 양호(임계값 위반 안 함)으로 처리 |

**알림 설정:**

| 항목 | 값 |
|------|-----|
| SNS 주제 | 새 주제 생성 |
| 주제 이름 | `High_CPU_Alarm` |
| 이메일 | 본인 이메일 입력 |

**EC2 작업 추가:** 경보 상태 → 이 인스턴스 재부팅

**경보 이름:** `High_CPU_Alarm`

> 📧 설정 후 이메일로 수신된 **Confirm subscription** 링크 클릭 필수!

### 9-4. CPU 부하 테스트

```bash
# EPEL 리포지토리 활성화
amazon-linux-extras install -y epel

# stress-ng 설치
yum install -y stress-ng

# CPU 부하 발생 (10분간)
stress-ng --cpu 1 --timeout 10m --metrics --times
```

## 10. 실습 과제

### 실습 1) EC2 삭제 후 재생성

1. 기존 EC2 삭제 후 인스턴스 목록 확인
2. 새 EC2 생성:

| 항목 | 값 |
|------|-----|
| 인스턴스 이름 | `계정-ec2` (예: `sgu-202500-ec2`) |
| 인스턴스 타입 | `t3.nano` |
| 키페어 | 기존에 생성한 pem 사용 |
| 보안 그룹 | 기존에 만든 SG 사용 |
| 태그 | Key: `class` / Value: `class-3a` 또는 `class-3b`, `class-3c` |

---

### 실습 2) 네트워크 경보 생성

**EC2에서 실행:**

```bash
# iperf3 설치
sudo yum install -y iperf3

# iperf3 서버 모드 실행
iperf3 -s &

# CPU + 네트워크 부하 동시 발생
iperf3 -s & sleep 2 && stress-ng --cpu 1 --timeout 10m --metrics --times & iperf3 -c 127.0.0.1 -t 600 -b 100M
```

**CloudWatch 경보 설정:**

| 항목 | 값 |
|------|-----|
| 지표 | `NetworkOut` |
| 임계값 | `NetworkOut >= 20000` |
| 경보 이름 | `High_Network_Alarm` |

**완료 후:** 인스턴스 재기동 → 이메일로 알람 수신 확인

## 📌 참고 사항

- 인스턴스 유형 변경 시 반드시 **중지 후** 진행
- 키페어 파일(`.pem` / `.ppk`)은 **재발급 불가** → 반드시 안전하게 보관
- Security Group은 인스턴스에 대한 **방화벽** 역할 → 필요한 포트만 허용
- CloudWatch 세부 모니터링 활성화 시 **추가 요금** 발생할 수 있음
- `terminated` 상태의 인스턴스는 **복구 불가**
