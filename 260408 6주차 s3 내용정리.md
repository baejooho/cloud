## S3란?

- AWS에서 **EC2와 함께 가장 오래되고 기본이 되는 서비스**
- 확장성, 내구성, 보안성이 뛰어난 **객체 스토리지 서비스**
- **99.999999999% (11 9's)** 의 내구성 제공 → 데이터 손실 최소화
- 데이터 저장 공간 **무제한**
- 전 세계 수백만 기업이 사용하는 핵심 스토리지 서비스

---

## S3 특징

| 특징 | 설명 |
|------|------|
| 객체 스토리지 | 파일 저장 전용 (EBS/EFS 등 블록 스토리지와 구분) → 어플리케이션 설치 불가 |
| 글로벌 서비스 | 서비스 자체는 글로벌, 데이터는 특정 리전에 저장 |
| 무제한 용량 | 단일 객체 0byte ~ 5TB 저장 가능 |
| 고가용성 | 최소 3개 이상의 물리적으로 분리된 AZ에 데이터 복제 |
| 간편한 접근 | OS 도움 없이 객체별 직접 접근 가능 |

## 버킷과 객체

### 버킷 (Bucket)
> S3에서 저장공간을 구분하는 단위 (디렉터리 개념)

- **전 세계 고유 이름** 필요 → 리전과 무관하게 중복 불가
- 한번 생성 시 **이름 변경 불가**

### 객체 (Object)
> S3에 저장되는 기본 데이터 단위

| 구성요소 | 설명 |
|----------|------|
| Owner | 소유자 정보 |
| Key | 객체 이름 (경로 역할) |
| Value | 파일 데이터 자체 |
| Version ID | 파일 버전 식별자 (버전 관리 활성화 시에만 생성) |
| Metadata | 파일 관련 정보 (기본 + 사용자 정의 가능) |

#### 객체 예시
```
S3 URI  : s3://sgu-202500-s3/test/haeri.txt
ARN     : arn:aws:s3:::sgu-202500-s3/test/haeri.txt
Key     : test/haeri.txt
Region  : ap-northeast-2 (서울)
```

## 내구성 & 가용성

- **내구성**: 파일을 잃어버리지 않을 능력
- **가용성**: 내가 원할 때 그 파일을 쓸 수 있는 능력

| 항목 | 수치 |
|------|------|
| 내구성 | 99.999999999% (S3 Standard) |
| 가용성 SLA | 99.99% (S3 Standard) |

> **SLA (Service Level Agreement)**: 서비스 수준 협약 — "이 서비스를 얼마나 안정적으로 사용할 수 있는가"

### 가용성별 연간 다운타임

| 가용성 | 연간 다운타임 |
|--------|--------------|
| 99.99% | 약 **52분** 이하 |
| 99.9%  | 약 **8시간 45분** 이하 |
| 99.5%  | 약 **1일 18시간** 이하 |

## 스토리지 클래스

> 저장 목적과 예산에 따라 다른 클래스를 선택

| 클래스 | 내구성 | 가용성 | AZ 수 | 최소 보관 기간 | 주요 사용 사례 | 고려사항 |
|--------|--------|--------|--------|----------------|----------------|----------|
| **S3 Standard** | 11 9's | 99.99% | 3+ | 없음 | 자주 액세스하는 데이터, 빅데이터 분석 | - |
| **S3 Intelligent-Tiering** | 11 9's | 99.99% | 3+ | 없음 | 액세스 패턴이 예측 불가한 데이터 | 객체별 패턴 모니터링, 자동 요금 최적화 |
| **S3 Standard-IA** | 11 9's | 99.99% | 3+ | 30일 | 수명이 길고 자주 액세스하지 않는 데이터 | 검색 비용 발생 |
| **S3 One Zone-IA** | 99.999999% | 99.95% | 1 | 30일 | 중요도 낮고 자주 액세스하지 않는 데이터 | 단일 AZ, 낮은 내구성 |
| **S3 Glacier** | 11 9's | 99.99% (복원 후) | 3+ | 90일 | 장기 보관 데이터 (백업, 아카이브) | 즉시 액세스 불가, 복원 시 지연 |
| **S3 Glacier Flexible** | 11 9's | 99.99% (복원 후) | 3+ | 90일 | 장애 복구용, 백업 데이터 | 분~시간 단위 복원 시간 소요 |
| **S3 Glacier Deep Archive** | 11 9's | 99.99% (복원 후) | 3+ | 90일 | 법적 보관 의무 데이터 | 복원까지 최대 12시간 이상 |
| **S3 on Outposts** | - | - | 온프레미스 | - | 온프레미스 환경 데이터 저장 | 로컬 환경, S3 SDK & IAM 연동 필요 |

### 클래스 선택 기준 (비용 vs 접근 빈도)

```
높은 가격 ──────────────────────────────────► 낮은 가격
파일저장

S3 Standard → S3-IA → S3-Intelligent → S3-One Zone IA
→ S3 Glacier Instant → S3 Glacier Flexible → S3 Glacier Deep Archive → S3 on Outposts

자주 접근 ◄──────────────────────────────── 거의 접근 안 함
```

## S3 보안

### 기본 원칙
- 기본적으로 **모든 객체는 Private(비공개)** 상태로 생성됨
- 퍼블릭 접근은 **명시적으로 허용**해야 함 (예: 웹 호스팅)

### 접근 제어 방식

#### 1. Bucket Policy (버킷 정책)
- **버킷 단위**로 적용되는 리소스 기반 정책
- 언제, 어디서, 누가, 무엇을, 어떻게 할 수 있는지 정의
- 익명 사용자 또는 다른 계정에 대한 권한 부여도 가능

```json
{
  "Effect": "Allow",
  "Principal": "*",
  "Action": "s3:GetObject",
  "Resource": "arn:aws:s3:::demo.rwlecture.com/*"
}
```

> `arn:aws:s3:::my-bucket/images/*` → 해당 경로의 모든 객체 대상으로 권한 설정

#### 2. IAM 정책

| 유형 | 설명 |
|------|------|
| **Identity-based Policy** (자격 증명 기반) | IAM 사용자, 그룹, 역할에 부여 → "누가 무엇을 할 수 있는가" |
| **Resource-based Policy** (리소스 기반) | 리소스(S3, SQS 등)에 직접 부여 → "이 리소스에 누가 접근 가능한가" |
